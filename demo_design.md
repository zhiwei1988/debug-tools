# 10GB+ 超大 Tar.Gz 固件包前端直读演示系统设计方案

## 1. 架构概览

针对包含超大二进制文件（类似监控设备的高清视频流存储或庞大的升级固件）与微小标记文件的 `.tar.gz` 压缩包，传统的“先上传到后端服务器再解析”方案会产生极高的网络带宽成本，且上传 10GB 文件耗时极长。

本设计采用**纯前端浏览器本地解析方案**。利用 HTML5 File API、Web Worker 和流式解压缩/解析算法，直接在用户浏览器中按块读取磁盘文件并进行内存解析。这种方案不仅响应迅速，还能完美兼容 Chrome 50+ 及 360 安全浏览器 9+ 等较老版本的环境。

## 2. 核心挑战与解决策略

**挑战一：10GB 级别文件的内存溢出风险**
浏览器标签页的内存限制通常在 2GB 到 4GB 之间。直接将 10GB 文件加载进内存会导致浏览器崩溃。
**解决策略**：使用 `File.slice()` 和 `FileReader` 进行分块读取（例如每次读取 10MB），并通过管道送入解压算法。

**挑战二：Gzip 格式无法随机访问**
`.gz` 压缩格式不支持随机寻址，我们无法直接跳到压缩包尾部去读取标记文件。
**解决策略**：采用**流式边解压边丢弃**机制。无论是标记文件在前还是在后，均从头开始流式解压。遇到大二进制文件的数据块时，仅统计已解压的字节数而不做存储，解压后立刻释放内存；遇到标记文件时，才将其内容存入内存。

**挑战三：解析过程阻塞 UI 渲染**
解压 10GB 的 Gzip 数据属于 CPU 密集型任务，如果在主线程运行，会导致页面完全卡死。
**解决策略**：引入 **Web Worker** 技术。将分块读取、解压和 Tar 结构解析的逻辑全部放入独立的后台线程，主线程仅负责显示进度条和最终结果。

## 3. 详细设计



### 3.1 模块划分与数据流向

**UI 主线程模块**
负责渲染文件选择框、进度条和结果显示区域。用户选择文件后，将 `File` 对象的所有权转移给 Web Worker，并监听来自 Worker 的进度消息（解压进度、解析状态）和最终的标记文件内容。

**Web Worker 控制模块**
接收到 `File` 对象后，维护一个读取游标。使用 `FileReader` 以 ArrayBuffer 格式循环读取下一块数据（如 10MB）。读取完一块后，立即交给解压模块。

**Gzip 解压模块 (基于 pako.js 或 fflate)**
由于需要兼容较早版本的浏览器（不支持原生的 `DecompressionStream`），推荐引入支持 chunking（分块处理）的第三方库。该模块接收压缩块数据，输出解压后的原生 `.tar` 字节流，并触发回调将数据传给 Tar 解析器。

**Tar 解析状态机模块**
核心模块。维护一个内部缓冲区，用于拼接跨块的数据。Tar 文件的标准结构是：512 字节的文件头（包含文件名、文件大小） + 文件内容 + 补齐到 512 字节的 padding。

### 3.2 Tar 解析状态机逻辑设计

由于解压出来的流是连续的块，Tar 解析器需要设计为有限状态机（FSM）：

**状态一：READING_HEADER（读取文件头）**
等待内部缓冲区累积满 512 字节。读取到 512 字节后，解析前 100 字节获取文件名，解析第 124 字节开始的 12 字节（八进制字符串）获取文件大小。计算该文件需要的 padding 字节数：`Math.ceil(size / 512) * 512 - size`。判断当前文件是否为“标记文件”。切换至 READING_DATA 状态。

**状态二：READING_DATA（读取文件数据）**
根据上一步解析出的文件大小，从流中消费数据。
**如果当前文件是标记文件**：将消费的字节追加到标记文件内容缓冲区中。当内容读取完毕（可能需要跨越多个解压块），通过 `postMessage` 将文本内容发送给主线程。
**如果当前文件是大二进制文件**：仅将“需读取的剩余字节数”减去当前接收到的字节数。**不保存任何数据**。这使得 10GB 的数据会在内存中一闪而过被垃圾回收机制清理，确保内存占用极低。
数据读取完毕后，切换至 READING_PADDING 状态。

**状态三：READING_PADDING（跳过填充）**
根据文件头计算的 padding 长度，丢弃后续相应的字节。完成后切换回 READING_HEADER 状态，准备解析下一个文件。

### 3.3 应对文件布局差异

系统设计能够自适应标记文件在压缩包中的不同位置：

**场景 A：标记文件在最前面**
状态机在读取完前几个块时，就会解析出标记文件内容并发送给主线程。Web Worker 在发送成功后，可以**直接终止文件读取操作并自毁**（调用 `worker.terminate()`）。这样不需要解压后续的 10GB 数据，通常只需几毫秒到几秒即可完成，体验极佳。

**场景 B：标记文件在最后面**
状态机会在丢弃 10GB 二进制文件的过程中持续运行。虽然内存不会溢出，但解压 10GB 数据仍需要消耗一定的 CPU 时间（视客户端配置而定）。在此期间，Web Worker 可以每处理 100MB 数据向主线程发送一次进度通知，主线程更新进度条，安抚用户等待焦虑。一旦大文件跳过完毕，即可瞬间读出末尾的标记文件。

## 4. 技术选型与 API 使用建议

**前端框架**：原生 HTML/JS 即可，无需复杂框架，方便封装成独立组件。
**文件读取**：`Blob.prototype.slice(start, end)` 结合 `FileReader.readAsArrayBuffer()`。该 API 在 Chrome 6+、Firefox 4+ 中均可用，完美满足环境约束。
**解压缩库**：推荐使用 `fflate`（体积更小，性能更好）或 `pako`。使用其支持的 Push/Streaming 模式。例如 `pako.Inflate` 对象的 `onData` 回调。
**字符解码**：标记文件内容如果是纯文本，从 ArrayBuffer 还原为字符串可使用 `TextDecoder('utf-8')`（Chrome 38+ 支持），保证中文或特殊字符不乱码。
